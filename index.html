Static Test Fixtures are not fun

We stared writing tests within a large project and got trapped by the number of fixtures we had to create for each of them, multiplied by the amount of tables. We also got tired of having to declare long lists of fixtures in our integration tests, as well as maintaining them. Eventually, we could not understand our tests. This is something I'll illustrate in the following slide.

CakePHP fixtures are static

CakePHP Fixture Factories are dynamic

CakePHP fixtures are static, they are hard coded and shared across all tests, once loaded. Nicolas Masson came up with the idea to create fixture factories in order to generate test fixtures dynamically for CakePHP too. You create what you need inside your test. The first version proposed by Nico got developed and maintained by Juan Pablo. I will explain in a simple example what this all means.

Example
User permissions

UsersTable ↔ UsersGroupsTable ↔ PermissionsTable

5 Tables 


                            users
                            users_users_groups
                            users_groups
                            users_groups_permissions
                            permissions

For example, we ended up with a large amount of user scenarios with different permissions. Every new test made the reading of the fixtures ever more impossible. During this presentation, I will take the example of user permissions. We have a table users, users belong to many user groups, and user groups belongs to many permissions. We end up with 5 tables, and therefore 5 test fixture files. Let's say I need a user with permission Foo. I need to go to the permission fixtures and search for permission foo, note its id, search in the pivot table users groups permissions which user groups this permission is associated. Hopefully I find one. Knowing this or these user group ids, I need to search in the pivot table users users groups for one user belonging to the user groups. A terrible pain. There are ways to ease the pain, but the pain will always remain.

Static 


							public function testUserPermission()
							{
							   $user = $this->Users->get(45, [
						         'contain' => ['UsersGroups.Permissions']
						   ])

							   $act = $this->Users->hasPermission($user, 'Foo');

							   $this->assertTrue($act);
							}

This is a test consuming CakePHP traditional fixtures. Let's assume that the user 45 had permission Foo. If my test needs to act on user 45, I need to fetch it with a query based on its id . It's double work. Additionally, if you want to add an additional permission to user 45, things will keep growing, and the complexity will make you think twice before you write another test.

Dynamic 


							public function testUserPermission()
							{
							   $user = UserFactory::make()
								->with('UsersGroups.Permissions', ['name' => 'Foo'])
								->getEntity();

							   $act = $this->Users->hasPermission($user, 'Foo');

							   $this->assertTrue($act);
							}

Whereas in the case of dynamic fixtures, you create the user you need for this test within the test. We use here a user factory. The method with allows to insert associated data based on the associations defined in your tables. The following will create a user, belonging to whatever users group, belonging to a permission Foo.

Tests should be

Readable

Easy to write

Fast

Tests should meet the following criteria: they should be intelligible. Ideally you should be able to read them like prose.
They should be easy to write and to maintain. The excuse: no time for writing tests should not be legitimate. Autocompletion should assist you.
They should run fast. To that aim, avoid interaction with the DB and do not load unnecessary fixtures.

Why test?

  • Clean code

  • Documentation

  • Code integrity

  • Saves time

But why do we write tests? Tests force you to write clean code. If you struggle writing your test, it could be due to the complexity of the code your are testing
If your tests can easily be read and understood, they document your source code.
Whenever your code is refactored, features are added, or external packages get updated, tests ensure that your code retains it's integrity.
Tests significantly reduce the amount of time spent on code maintenance. It saves time and efforts.

Tests are the roots of your code

If your code is a tree, consider tests as its roots. It is its invisible part, it insures that your code grows stable, and does not get wiped out by the first storm.

Test anatomy

Arrange

Act

Assert

How do tests look like? Most of them will follow the famous AAA pattern.
Arrange will prepare the scenario of your test
Act will call the method under test
Assert will ensure that the result of the method meets expectations. Test driven development means that you write Arrange and Assert before Act.

Unit test


							public function testUserPermission()
							{
							  // Arrange
							   $user = UserFactory::make()
								->with('UsersGroups.Permissions', ['name' => 'Foo'])
								->getEntity();

							   // Act
							   $act = $this->Users->hasPermission($user, 'Foo');

							   // Assert
							   $this->assertTrue($act);
							}

This is how the test looks like with the factories. The facot

Unit tests


						public function dataProviderForUserWithPermissionFoo()
						{
						   return [
						      ['Admin', true],
						      ['Guru',  true],
						      ['Foo',   true],
						      ['Bar',   false],
						   ];
						}

						/** @dataProvider dataProviderForUserWithPermissionFoo */
						public function testUserHasPermissionFoo(
						   string $permission,
						   bool $expected
						)
						{
						   $user = UserFactory::make()
							->withPermission($permission)
							->getEntity();

						   $act = $this->Users->hasPermission($user, 'Foo');

						   $this->assertSame($expected, $act);
						}

Since creating fixtures are now made a piece of cake, we are not afraid any more to add tests to make our application more robust. For example here, we will test the hasPermission method against 4 users with different permissions. Admin and Gurus should have all permission Foo. A user with permission Bar not. The PHPUnit data provider tool enables to run one test with different settings. Here we pass the permission of the user created in the test, and the assert expectation. The test remains simple to read, and the test is complete.

Load the plugin


						   // In Application.php
						   ...
						   protected function bootstrapCli(): void
						   {
						      // Load more plugins here
						      $this->addPlugin('CakephpFixtureFactories');
						   }

Update the listeners in phpunit.xml 


							bin/cake fixture_factories_setup

The package uses an extension of the CakePHP native test listener. It is doing much less. I'll get back to it.

Bake your factories 


							bin/cake bake fixture_factory -a

Bake your factories. The plugin will crawl for all the tables and will generate a skeleton for your fixture factories. You can also specify one particular table, or a plugin, and more. This is described in the docs.

Migrations (optional) 

In tests/bootstrap.php


						\CakephpTestMigrator\Migrator::migrate();

The Migrator is one of the key fixtures of the plugin. It is actually not in the plugin, but aside. The idea is that we do not drop tables, at no time. The migrator will run migrations on Monday, and the schema of your test DB will evolve exactly the same way as your productive DB does. Migrations get indirectly tested. You do not have to care about any maintenance of your DB. If you ever switch branches, the migrator detects it and drops all the tables.

							namespace App\Test\Factory;

							use CakephpFixtureFactories\Factory\BaseFactory;
							use Faker\Generator;

							class UserFactory extends BaseFactory
							{
							   protected function getRootTableRegistryName(): string
							   {
								return "Users";
							   }

							   protected function setDefaultTemplate(): void
							   {
								$this->setDefaultData(function (Generator $faker) {
								   return [
									  'username' => $faker->userName
									  'password' => $faker->password,
									  'email'    => $faker->email,
								   ];
								});
							   }

							   public function withPermission(string $permission): self
							   {
								return $this->with('UserGroups.Permissions', [
								  'name' => $permission
								]);
							   }

							   public function admin(): self
							   {
								return $this->withPermission('Admin');
							   }

							   public function inFantasticMood(): self
							   {
								// Make this user happy here
								return $this;
							   }
							}

Test Life Cycle

tests/Bootstrap

Run migrations

Test Starts

Truncate Dirty Tables

The test life cycle using the listener made for the factories is slightly different from the one out of the box in CakePHP. The schema of the DB is created using the migrations, or running your Sql commands on your test DB. This documents your schema. Additionally, it involves the migrations in your test cycle. Your migrations get tested. This is very useful. Before every test, the dirty tables get truncate. The schema is never modified. This speeds up significantly the tests.

Performance

Time (Y axis) vs Number of tests (X axis)

->Pilot study here<-

The x-axis is the number of tests per TestClass. In y-axis is the time in second to perform the whole test suite. The numbers are taken form a performance study available here. In red, the fixtures generated by the factories are not persisted in the data base. This is real unit testing, or at least close to it, since there is no interaction with the database. Tests remain low. In blue sky, the fixtures are generated by the factories, and persisted. This takes significantly more time time. The execution time is proportional to the size of your app. In dark blue, the fixtures are generated by the CakePHP native fixtures. Since all fixtures of a fixture file are loaded for a given test, the more your application grows, the faster it gets slower. The law is hyperbolic. You will probably not achieve the red line. But if you keep in mind that avoiding DB interaction speeds up your test, and your app in general, the fixture factories will enable you to write and test methods that rather use entities than ids. The numbers and the test setting are available on a github repo here.
